# Multiuser Google OAuth Design

Date: 2026-05-19

## Goal

Expand Personal Hermes from a single-user Telegram assistant into a multiuser assistant where each Telegram user can connect their own Google account and use the bot against their own Gmail and Google Calendar.

The recommended product shape is Telegram-first onboarding with a small web OAuth callback. Telegram remains the primary interface. The web surface exists only to complete Google sign-in, consent, and account connection.

## Scope

### In Scope

- Telegram `/connect` onboarding for new users.
- Google OAuth consent for Gmail and Calendar access.
- User-specific Google token storage.
- User-specific Gmail polling, calendar reminders, daily agendas, pending replies, and audit logs.
- Minimal OAuth web server for authorization callback and success/error pages.
- `/status` and `/disconnect` bot commands.
- Invite-only beta controls for the first rollout.
- Encryption at rest for Google refresh tokens.
- Per-user scheduler isolation so one user's failures do not block other users.

### Out of Scope

- Full account dashboard.
- Billing.
- Admin web UI.
- Multiple Google accounts per Telegram user.
- Team/shared mailbox workflows.
- Public self-serve launch.
- Telegram or Google webhooks.
- Automatic email sending without explicit Telegram approval.

## Recommended Approach

Use Telegram-first onboarding with a small OAuth web runtime.

The user starts in Telegram, receives a Google authorization link, approves access in the browser, and returns to Telegram after the account is connected. This matches the product because Telegram is the main user interface, while still following Google's browser-based OAuth requirements.

This approach is lighter than building a full web app and more usable than manual provisioning. It also gives the application a clean identity mapping:

```text
telegram_user_id -> app user -> google account tokens
```

## Product Flow

1. User opens the Telegram bot.
2. User sends `/connect`.
3. Bot creates a short-lived onboarding session tied to the Telegram user and chat.
4. Bot sends a Google authorization URL.
5. User approves Gmail and Calendar scopes in the browser.
6. OAuth callback validates state, exchanges the code for tokens, and stores the Google account against the user.
7. Browser shows a simple success page.
8. Bot sends a confirmation message in Telegram.
9. User can use Gmail and Calendar features through Telegram.

If a connected user sends `/connect` again, the bot should report the current connection and offer reconnect only when useful.

## Architecture

The current Python service remains the foundation, but it gains three runtime responsibilities.

### Telegram Runtime

The Telegram runtime handles bot polling, commands, inline buttons, and replies.

Instead of checking a single configured `TELEGRAM_AUTHORIZED_USER_ID`, inbound updates are resolved to an application user by Telegram user ID and chat ID. Unknown users are directed to `/connect` if invite policy allows them, or told that access is invite-only.

### OAuth Web Runtime

The OAuth web runtime is a small HTTP server with:

- `GET /oauth/google/callback`
- simple success and error pages

The authorization URL can be generated by the Telegram command handler rather than exposed as a public start endpoint. The callback owns OAuth state validation, code exchange, token storage, and post-connect Telegram confirmation.

### User-Scoped Scheduler

The scheduler runs jobs across active connected users.

For each user, it loads that user's Google account tokens, builds a Google client for that user, and executes Gmail and Calendar jobs with a `user_id`. Failures are recorded per user and do not stop jobs for other users.

## Components

- `UserStore`: creates and resolves users by Telegram identity, stores status, and enforces invite policy.
- `OAuthSessionStore`: creates short-lived state records and marks them used after callback completion.
- `GoogleAccountStore`: stores Google account identity, encrypted tokens, scopes, expiry, and connection status.
- `TokenCipher`: encrypts and decrypts Google tokens using an app-level secret key.
- `GoogleAccountProvider`: returns usable Google credentials for a specific user and refreshes tokens when needed.
- `AssistantRouter`: resolves inbound Telegram updates to a user and routes commands/actions in user scope.
- `MailService`: processes Gmail messages and pending replies for one user at a time.
- `CalendarService`: handles availability, agenda, and reminders for one user at a time.
- `Scheduler`: iterates over active connected users and runs isolated user jobs.

## Data Model

New tables:

### `users`

- `id`
- `telegram_user_id`
- `telegram_chat_id`
- `display_name`
- `username`
- `status`: `pending`, `active`, `revoked`, `disabled`
- `created_at`
- `updated_at`

### `oauth_sessions`

- `state`
- `telegram_user_id`
- `telegram_chat_id`
- `expires_at`
- `used_at`
- `created_at`

### `google_accounts`

- `user_id`
- `google_subject`
- `google_email`
- `encrypted_access_token`
- `encrypted_refresh_token`
- `granted_scopes`
- `token_expires_at`
- `status`: `active`, `reauth_required`, `revoked`
- `created_at`
- `updated_at`

Existing tables must gain `user_id`:

- `seen_emails`
- `pending_replies`
- `reply_audit_log`
- `calendar_agenda_notifications`
- `calendar_reminders`
- `conversation_state`

Uniqueness and deduplication should become user-scoped. For example, `seen_emails` should deduplicate by `(user_id, email_id)`, and calendar reminders should deduplicate by `(user_id, event_id, event_start_at)`.

## OAuth And Google Access

OAuth sessions use random state values that are short-lived and single-use. The state is tied to the Telegram user and chat that requested `/connect`.

The app requests only the scopes needed for the current product:

- Gmail read access for inbox notifications and summaries.
- Gmail send or modify access for approved replies and optional mark-read behavior.
- Calendar read access for agenda, reminders, and availability.

The exact Google scope strings should be selected during implementation against the Google API client being used.

Gmail scopes can be sensitive or restricted. For the first rollout, the application should operate as an invite-only beta using configured Google OAuth test users. A public launch will likely require Google OAuth app verification.

## Security Rules

- A Telegram user can only act on records tied to their own `user_id`.
- OAuth state values are random, short-lived, and single-use.
- Google refresh tokens are encrypted at rest with an app-level encryption key from environment or secret storage.
- Callback payloads and Telegram callback data are not trusted as authority; every action is resolved through stored records.
- A disconnected, revoked, disabled, or reauth-required account cannot run background jobs or send email replies.
- Pending reply send actions fail closed if ownership, status, expiry, or Google account state is invalid.
- Full email bodies are not logged unless debug logging is explicitly enabled.
- Every sent reply is recorded in the audit log with `user_id`.

## Telegram Commands

- `/connect`: creates an OAuth session and sends the Google authorization link.
- `/status`: reports whether the Telegram user is connected, disconnected, or needs reauthorization.
- `/disconnect`: revokes local access, disables background jobs, and removes or invalidates stored Google tokens.

Unknown commands from connected users can keep the current assistant behavior. Unknown commands from unconnected users should explain how to connect or that the beta is invite-only.

## Failure Handling

- Expired OAuth sessions tell the user to run `/connect` again.
- Reused OAuth state values are rejected.
- OAuth success followed by Telegram send failure still leaves the user connected; the next Telegram interaction can report status.
- Token refresh failure marks the Google account `reauth_required` and sends a reconnect prompt.
- Per-user Gmail or Calendar failures are logged against that user and do not stop the scheduler globally.
- Background jobs skip users without an active Google account.
- Approved reply sends fail closed when the account is disconnected, revoked, expired, or owned by another user.

## Migration Strategy

The current single-user configuration can be used as a bootstrap path during migration.

Recommended migration:

1. Add user and Google account tables.
2. Add `user_id` columns to existing state tables.
3. Create a bootstrap user from the current Telegram authorized user/chat configuration if existing state exists.
4. Backfill existing rows to the bootstrap user.
5. Update services to require `user_id`.
6. Remove reliance on global authorized Telegram config after multiuser resolution is stable.

## Rollout Plan

1. Add multiuser data model and user resolution while preserving the existing single-user config as a bootstrap fallback.
2. Add Telegram `/connect` and the OAuth callback runtime.
3. Migrate mail and calendar services to user-scoped execution.
4. Add the per-user scheduler loop.
5. Add `/status` and `/disconnect`.
6. Run invite-only beta with a small allowlist.
7. Later, consider public onboarding, admin tools, billing, and webhook upgrades.

## Testing

Unit tests:

- User creation and lookup by Telegram identity.
- Invite-only access decisions.
- OAuth session creation, expiry, reuse rejection, and Telegram identity binding.
- Token encryption and decryption.
- Google account status transitions.
- User-scoped Gmail deduplication.
- User-scoped calendar reminder deduplication.
- Pending reply ownership enforcement.
- `/status` and `/disconnect` command behavior.

Integration-style tests with mocks:

- `/connect` creates an OAuth session and sends an auth URL.
- OAuth callback stores Google account tokens and confirms in Telegram.
- Gmail polling processes only the target user's mailbox.
- Calendar reminders are sent only to the owning user's Telegram chat.
- Token refresh failure moves a user to `reauth_required`.
- One user's scheduler failure does not prevent another user's jobs from running.

## Implementation Decisions

- Use direct Google API OAuth clients for the multiuser path. The current single-user `gog` access can remain during migration, but user-owned OAuth grants should not depend on a process-level Google account.
- Choose the exact Google OAuth scope strings during implementation from current Google API documentation, using the least-privilege scopes that support Gmail inbox reads, approved Gmail sends, optional mark-read behavior, and Calendar reads.
- `/disconnect` should invalidate local tokens immediately and attempt token revocation at Google. Local disconnect succeeds even if remote revocation fails, and the failure is logged for follow-up.
- Store `GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_CLIENT_SECRET`, OAuth redirect URL, and the token encryption key in environment or server secret storage. None of these values are committed.
